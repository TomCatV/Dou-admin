# Dou 小圈平台端营收与手续费设计

最后更新：2026-05-29

## 1. 目标与结论

本文用于定义 Dou 小圈平台端如何在圈主商业后台、H5 商品页、微信/支付宝扫码支付、钱包结算、提现、售后退款和后续运营工具中获得可持续收入。

核心结论：

1. 第一阶段继续采用“平台统一收款 -> Dou 账务拆分 -> 圈主钱包 -> 圈主提现”的模式，不做圈主直清和服务商分账。
2. 平台收入不要只设计成“支付手续费”。Dou 应拆成“交易技术服务费 + SaaS 套餐 + 增值工具 + 分销/营销工具 + 人工服务”的组合。
3. 第一版交易技术服务费建议用统一费率起步：普通圈主 `20%`，后续随套餐降档到 `15% / 10% / 6%`，重点鼓励高质量圈主升级套餐和长期经营。
4. 支付通道成本、退款手续费损耗、商家转账出款成本、内容审核成本和客服风控成本必须进入平台毛利测算，不能只看订单抽佣。
5. 所有费率、减免、结算和退款冲正必须可配置、可审计、可对账，后台不能直接把资金终态改成成功。

## 2. 对标调研：链动小铺怎么赚钱

### 2.1 可确认能力

本地方案文档已记录《链动小铺平台商家开店指引（新版）》中的核心能力：快速注册开店、支付宝/微信收款通道、店铺设置、独立店铺链接、商品分类、商品发布、卡密库存、订单管理、售后投诉、主动退款、钱包资金、微信通知、代理分销、店铺 DIY。

公开资料能确认链动小铺定位是数字商品/卡密/虚拟资源交易平台，面向卡密、知识内容、资源下载等场景，强调低门槛开店、支付通道、自动发货、订单和售后闭环。但公开网页没有稳定披露统一的手续费标准，不同入口和版本可能存在会员套餐、商户等级、通道模式差异，因此本文不把具体数字当作事实引用。

本次调研到的公开资料口径存在明显差异，只能作为商业模式样本：

| 公开资料口径                                                                                               | 参考价值                                              |
| ---------------------------------------------------------------------------------------------------------- | ----------------------------------------------------- |
| 有资料称传统发卡网常见“月费 + 交易手续费”，费率约 `1%-5%`，链动小铺偏“阶梯式月费 + 较低甚至为零交易手续费” | 说明链动小铺可能更重套餐收入，而不是只靠订单抽佣      |
| 有资料称链动小铺或同类发卡平台会收 `2%`、`4%` 平台服务费，或提现收 `1%-3%`                                 | 说明同类市场接受按成交或提现收费，但口径不统一        |
| 有资料估算第三方聚合支付通道成本约 `0.38%-0.6%`，提现手续费约 `0.1%-1%`                                    | 可作为 Dou 做毛利测算的成本参考，不可当作真实合同费率 |

结论：链动小铺的关键不是某个固定手续费数字，而是把“开店工具、支付通道、自动交付、钱包提现、分销和高级功能”组合成收费体系。Dou 应借鉴这种组合收入结构，但费率必须由我们自己的成本、转化率和圈主承受能力决定。

### 2.2 业务模式推断

结合本地资料和同类发卡/虚拟商品平台的常见做法，链动小铺类平台的收入一般来自：

| 收入项           | 典型做法                                           | 对 Dou 的启发                                            |
| ---------------- | -------------------------------------------------- | -------------------------------------------------------- |
| 开店/会员费      | 免费入门，高级版收月费/年费                        | Dou 可用免费试用 + 基础版 + 专业版 + 品牌版              |
| 交易服务费       | 每笔成交按比例收取，或套餐越高费率越低             | Dou 第一阶段用平台技术服务费，默认 20%，套餐可降费率     |
| 支付通道成本差额 | 商户自配通道或平台聚合收款时，成本和收费之间有差额 | Dou 暂不对圈主透明拆通道成本，统一纳入服务费毛利         |
| 提现/出款成本    | 提现低于阈值可能收固定费用或限制频次               | Dou 建议先不单独收提现费，用最低提现金额和免费次数控成本 |
| 营销/分销工具    | 代理分销、优惠活动、通知、店铺装修收费             | Dou P4 可做增值模块，而不是第一版就复杂化                |
| 人工服务         | 审核、客服、售后仲裁、定制配置                     | Dou 平台可对企业/品牌圈主收人工服务费                    |

### 2.3 Dou 不应照搬的地方

1. 不照搬“纯发卡站”定位。Dou 的长期价值是圈子沉淀、买家复购、内容经营和 AI 助手。
2. 不先做圈主自配通道。第一阶段统一平台收款能更快做好回调验签、售后、退款、钱包和对账。
3. 不把手续费设计成隐藏扣费。圈主后台要展示“订单金额、平台服务费、预计入账、结算期、退款冲正规则”。
4. 不靠高提现手续费盈利。提现费容易伤害信任，第一版更适合用交易服务费和套餐定价覆盖出款成本。

## 3. Dou 当前业务基础

### 3.1 已有链路

Dou 当前已具备：

1. 商品与 H5 下单：资源卡升级为商品中心，支持公开店铺页、商品页和订单草稿。
2. 扫码支付：P2 已接入微信 Native 与支付宝当面付，支付成功由可信回调或查单推进。
3. 自动交付：支付成功后交付资源链接、提取码、兑换说明或卡密。
4. 钱包结算：`order_settlements` 记录订单毛额、平台服务费、圈主入账金额。
5. 提现：圈主钱包可提现，出款走微信商家转账到零钱。
6. 售后退款：已有资源卡售后与微信原路退款基础。

### 3.2 当前费率实现差异

历史设计文档写过 `CREATOR_PLATFORM_FEE_BPS=2000`，但当前后端实现使用的是：

```text
CREATOR_PLATFORM_FEE_RATE=0.20
```

计算逻辑：

```text
gross_amount = order.amount
platform_fee_amount = floor(gross_amount * CREATOR_PLATFORM_FEE_RATE)
creator_amount = gross_amount - platform_fee_amount
```

后续建议统一改成基点制配置，例如 `PLATFORM_TRADE_FEE_BPS=2000`，避免小数配置在多语言、多环境和后台表单里出现精度歧义。

## 4. 平台端营收模型

### 4.1 收入分层

| 层级              | 收入项                                       | 第一版建议                          |
| ----------------- | -------------------------------------------- | ----------------------------------- |
| L1 交易技术服务费 | 每笔已支付订单按比例扣除                     | 先做，默认 20%                      |
| L2 SaaS 套餐      | 按月/年收取后台能力费用                      | 已有租户套餐基础，下一阶段强化      |
| L3 增值工具       | AI 经营助手、短信/通知、活动、装修、数据分析 | P4/P5 做                            |
| L4 分销与渠道服务 | 代理链接、渠道归因、佣金结算                 | P4 做，平台可对分销成交额加收服务费 |
| L5 人工服务       | 平台代配置、审核加急、售后仲裁、企业定制     | 手动收款和合同化，先不自动化        |

### 4.2 第一版收费方案

建议上线时采用“免费开店 + 成交抽取平台技术服务费”的低门槛模式：

| 圈主等级 | 适用对象         | 平台服务费 | 月费/年费 | 能力边界                         |
| -------- | ---------------- | ---------- | --------- | -------------------------------- |
| 体验版   | 新手、小规模试卖 | 20%        | 0 元      | 商品数量、月订单额、AI 次数有限  |
| 基础版   | 稳定小卖家       | 15%        | 待定      | 提高商品数、订单额、通知额度     |
| 专业版   | 有固定流量的圈主 | 10%        | 待定      | 更低费率、营销工具、数据看板     |
| 品牌版   | 高流水或团队经营 | 6% 起      | 合同价    | 专属配置、人工服务、可能支持直清 |

第一版如果暂不做套餐付费，也可以先只启用体验版统一 20%，但数据模型要预留 `fee_policy_id`、`tenant_plan_id` 和订单级快照，避免后续改费率影响历史订单。

### 4.3 费率不要低于成本线

平台实际毛利公式：

```text
平台毛利 = 平台服务费
        - 支付通道成本
        - 退款损耗
        - 提现/出款成本
        - 内容审核成本
        - 存储和流量成本
        - 客服和人工风控成本
        - 营销补贴和优惠券平台承担部分
```

因此平台看板必须同时展示：

1. GMV。
2. 平台服务费收入。
3. 圈主应结算金额。
4. 退款冲正金额。
5. 通道成本估算。
6. 出款成本估算。
7. 净收入估算。

## 5. 手续费命名和展示

### 5.1 对外命名

面向圈主和买家，建议统一叫“平台技术服务费”，不要叫“手续费”或“抽佣”。

展示文案：

```text
平台技术服务费用于支付收款通道、订单系统、自动交付、售后风控、内容审核、买家通知和平台运维成本。
```

### 5.2 买家侧展示

第一版建议买家侧商品价格就是实付价，不额外展示平台服务费。服务费由圈主侧承担，从订单成交额中扣除。

不建议第一版让买家额外支付手续费，原因：

1. H5 转化链路短，额外手续费会明显降低付款率。
2. 数字商品和资源卡价格通常较低，按笔加收容易显得“算计”。
3. 平台服务费本质是圈主使用 Dou 经营系统的成本，应在圈主后台透明展示。

### 5.3 圈主侧展示

订单详情展示：

```text
订单金额：¥100.00
平台技术服务费：¥20.00
预计入账：¥80.00
结算状态：待结算 / 可提现 / 提现中 / 已提现 / 已退款冲正
```

钱包页展示：

```text
成交收入：¥100.00
平台服务费：¥20.00
圈主收入：¥80.00
待结算：¥80.00
可提现：¥0.00
```

## 6. 资金和对账规则

### 6.1 订单维度快照

每笔订单支付成功时必须固化：

| 字段                  | 说明                             |
| --------------------- | -------------------------------- |
| `gross_amount`        | 买家实际支付金额                 |
| `platform_fee_amount` | 平台服务费                       |
| `creator_amount`      | 圈主入账金额                     |
| `fee_rate_bps`        | 订单当时费率                     |
| `fee_policy_id`       | 费率策略                         |
| `tenant_plan_id`      | 触发该费率的套餐                 |
| `channel_cost_amount` | 支付通道成本估算，后续对账可修正 |
| `platform_net_amount` | 平台净收入估算                   |

历史订单必须按订单快照结算，不能因圈主套餐升级或平台改费率而重算历史收益。

### 6.2 退款冲正

退款时按“先冲订单、再冲结算、再冲钱包”的顺序处理：

```text
退款成功
-> 订单进入 refunded / partial_refunded
-> 结算单按退款金额比例冲正
-> 未提现收益直接从 pending / available 扣回
-> 已提现收益进入圈主负余额或平台垫付策略
-> 平台服务费按退款比例冲减收入
```

第一版建议只支持全额退款；部分退款放到 P3 增强。

全额退款示例：

```text
订单 ¥100.00
平台服务费 20%，平台收入 ¥20.00，圈主收入 ¥80.00
退款成功后：
平台服务费收入 -¥20.00
圈主收入 -¥80.00
订单交付权限关闭，卡密禁用
```

### 6.3 已提现后退款

已提现订单退款是资金风险点。建议策略：

| 策略         | 第一版建议       | 说明                           |
| ------------ | ---------------- | ------------------------------ |
| 平台垫付     | 可作为兜底       | 用户体验好，但平台承担资金风险 |
| 圈主负余额   | 推荐作为正式策略 | 后续收入优先抵扣负余额         |
| 人工线下处理 | 大额或争议单使用 | 平台财务介入                   |

第一版默认：

1. 已提现后发生退款，平台先垫付给买家。
2. 圈主钱包形成 `negative_balance_amount`。
3. 圈主后续新订单收益先抵扣负余额，再转为可提现。
4. 负余额超过阈值时暂停提现和商品上架。

### 6.4 提现成本

第一版不单独向圈主收提现手续费。通过以下方式控成本：

1. 最低提现金额生产建议 `10 元` 或 `30 元`。
2. 单日免费提现次数限制，例如每天 1 次。
3. 小额高频提现进入冷却或人工审核。
4. 高风险圈主关闭自动审核。

如后续要收提现费，建议只对超出免费次数或低于阈值的提现收固定费用，且必须在提交前明确展示。

## 7. 费率策略设计

### 7.1 策略优先级

费率按以下优先级命中：

```text
订单专属费率
> 圈主临时活动费率
> 租户套餐费率
> 商品类目费率
> 平台默认费率
```

### 7.2 费率策略表建议

后续新增 `platform_fee_policies`：

| 字段                        | 说明                                                      |
| --------------------------- | --------------------------------------------------------- |
| `id`                        | 策略 ID                                                   |
| `name`                      | 策略名称                                                  |
| `scope_type`                | `global` / `tenant` / `category` / `product` / `campaign` |
| `scope_id`                  | 对应对象 ID                                               |
| `fee_bps`                   | 平台服务费基点                                            |
| `min_fee_amount`            | 最低服务费，单位分                                        |
| `max_fee_amount`            | 最高服务费，单位分，可为空                                |
| `starts_at` / `ends_at`     | 生效时间                                                  |
| `status`                    | `active` / `paused` / `expired`                           |
| `created_by_admin_id`       | 创建人                                                    |
| `created_at` / `updated_at` | 时间                                                      |

### 7.3 收入流水表建议

新增 `platform_revenue_ledger`：

| 字段                  | 说明                                                              |
| --------------------- | ----------------------------------------------------------------- |
| `id`                  | 流水 ID                                                           |
| `business_type`       | `order_fee` / `refund_reverse` / `subscription` / `manual_adjust` |
| `business_id`         | 订单、退款、套餐订单或调整单 ID                                   |
| `circle_id`           | 圈子                                                              |
| `tenant_id`           | 租户                                                              |
| `gross_amount`        | 原始成交额                                                        |
| `platform_fee_amount` | 平台服务费                                                        |
| `channel_cost_amount` | 通道成本                                                          |
| `net_revenue_amount`  | 净收入                                                            |
| `status`              | `booked` / `reversed` / `settled`                                 |
| `evidence_json`       | 第三方单号、费率策略、对账摘要                                    |
| `created_at`          | 时间                                                              |

平台收入也必须走流水，不允许只靠订单表聚合。

## 9. 权限和审计

| 操作             | 所需权限                    | 审计           |
| ---------------- | --------------------------- | -------------- |
| 查看平台营收     | `finance:revenue:view`      | 否             |
| 修改全局费率     | `finance:fee-policy:manage` | 是             |
| 修改单个租户费率 | `finance:fee-policy:manage` | 是             |
| 创建人工调整流水 | `finance:revenue:adjust`    | 是，需二次确认 |
| 导出收入流水     | `finance:revenue:export`    | 是             |
| 查看圈主服务费   | `tenant:wallet:view`        | 否             |

高风险动作要求：

1. 超级管理员或财务管理员才可修改费率。
2. 费率下调、免佣、人工调整必须填写原因。
3. 费率策略变更只影响未来订单，不影响历史订单。
4. 所有人工调整必须生成 `platform_revenue_ledger` 和 `admin_audit_logs`。

## 10. 配置建议

### 10.1 环境变量

第一版可先用环境变量兜底：

```text
PLATFORM_TRADE_FEE_BPS=2000
PLATFORM_TRADE_FEE_MIN_AMOUNT=0
PLATFORM_TRADE_FEE_BUYER_PAY=false
PLATFORM_WITHDRAW_FEE_ENABLED=false
PLATFORM_WITHDRAW_FREE_DAILY_COUNT=1
PLATFORM_WITHDRAW_LOW_AMOUNT_FEE=0
PLATFORM_NEGATIVE_BALANCE_BLOCK_AMOUNT=5000
```

现有 `CREATOR_PLATFORM_FEE_RATE=0.20` 可继续兼容，但新代码优先读取 `PLATFORM_TRADE_FEE_BPS`。

### 10.2 生产默认值

| 参数               | 建议                      |
| ------------------ | ------------------------- |
| 默认平台服务费     | 20%                       |
| 最低提现金额       | 10 元起，正式可调到 30 元 |

1. 人工调整不释放余额、不重算结算、不触发退款、不同步提现、不调用第三方支付接口。
2. 如果线上发现误操作，使用反向调整抵消，不允许删除历史流水。

为避免“代码已推送”与“线上已可用”混淆，Admin 仓库新增 `scripts/verify-finance-online.mjs`，通过 `corepack pnpm verify:finance-online` 执行线上财务模块烟测。

默认检查：

1. `ADMIN_BASE_URL`，默认 `https://admin.doucatapp.top`，首页构建产物必须包含 `人工调整`、`确认人工调整`、`确认反向调整`。
2. `API_BASE_URL`，默认 `https://api.doucatapp.top`，`GET /api/admin/finance/revenue/summary` 必须不是 404/405。
3. `POST /api/admin/finance/revenue/adjustments` 必须不是 404/405，且匿名模式应被 401/403 鉴权拦截。

带凭证检查：

1. 设置 `ADMIN_AUTH_TOKEN` 后，平台营收 summary 必须返回 200。
2. 设置 `ADMIN_AUTH_TOKEN` 后，人工调整接口使用空参数应进入 400/422 参数校验，而不是 401/403/404。
3. 设置 `ADMIN_AUTH_TOKEN` 后，`/api/admin/auth/me` 返回内容必须包含 `finance:revenue:adjust`。
4. 设置 `ADMIN_L2_AUTH_TOKEN` 后，人工调整接口必须返回 403，用于确认 2 级管理员没有调整权限。

完整上线门槛：

1. GitHub Actions 或等效部署日志确认 Dou-Server `master` 和 Dou-Admin `main` 均发布成功。
2. 生产 `.env` 若需非默认限额，应配置 `PLATFORM_REVENUE_ADJUST_MAX_AMOUNT`；不配置时默认 100000 分。
3. `corepack pnpm verify:finance-online` 在当前线上域名通过。
4. 用超管/1 级管理员完成一次小额人工调整和反向调整，确认收入汇总抵消、流水保留、审计日志完整。
5. 用 2 级管理员确认无人工调整入口或接口返回 403。
6. 真实支付闭环仍需商户参数、证书路径、回调域名和小额订单完成微信/支付宝支付、异步通知、查单和交付回归。
