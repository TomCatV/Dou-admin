# H5 扫码支付刷新与取消体验修复设计

最后更新：2026-06-09

## 背景

线上 H5 扫码支付页出现五个体验问题：自动刷新会让页面按钮和二维码区域闪烁；“取消支付”点击后用户感知不明确；取消后重新生成二维码被后端返回“订单当前不可支付”；支付方式默认顺序仍偏向微信；支付宝立即购买缺少足够清晰的 loading 和防重复点击。

## 官方契约

1. 微信 Native 支付由服务端创建 Native 支付订单并返回 `code_url`，商户页面展示二维码；支付结果以支付通知或查单为准。
2. 微信关闭订单和支付宝交易关闭都只代表第三方待支付交易关闭，不等同于 Dou 本地未付款订单永久取消。
3. 支付宝 `alipay.trade.precreate` 返回 `qr_code`，适合站内渲染支付宝扫码二维码；`alipay.trade.query` 用于兜底查单，`alipay.trade.close` 用于关闭当前交易。

## 本轮目标

1. 自动轮询使用静默状态，不改变主要按钮文案，不清空二维码，不触发全页 loading。
2. “取消支付”关闭当前支付意图，停止轮询，并在二维码区域展示“已关闭”，但不把本地未付款订单改为不可支付。
3. “重新生成二维码”直接创建新的支付意图，不再要求用户先清空再二次点击。
4. 商品页和支付页默认展示支付宝在前，默认选中支付宝；只有支付宝不可用时回落微信。
5. 商品页“立即购买”和支付页“生成二维码”统一使用 loading、防重复点击和 `aria-busy`。

## 前端方案

1. `checkout.vue` 将自动轮询与手动刷新拆为两个状态：静默轮询不驱动按钮 loading；手动点击刷新才显示“正在刷新”。
2. 轮询只更新 `order` 与 `paymentIntent` 状态，不重置 `qrDataUrl`；只有创建新支付意图或明确清空页面时才重新渲染二维码。
3. `canClosePayment` 仅在 `created`、`qr_issued`、`unknown` 等非终态展示“取消支付”，关闭后隐藏该按钮。
4. `regeneratePaymentIntent()` 会保留当前通道并直接请求后端创建新支付意图，成功后重启倒计时和静默轮询。
5. `product.vue` 默认选中 `alipay_precreate`，支付方式按钮顺序调整为支付宝、微信；支付宝购买主链路默认创建站内二维码支付意图，不再默认 `mode=cashier` 外跳。

## 后端协同

1. `closePaymentIntent` 只关闭当前 `commerce_payment_intents`，不再把 `orders.status` 改为 `closed`。
2. 第三方查单返回 `CLOSED` / `TRADE_CLOSED` 时同样只关闭支付意图，订单保持 `created`，买家可重新生成二维码。
3. 为兼容旧逻辑已经误关闭的 H5 订单，创建新支付意图时若订单来自 `commerce_order_drafts`、没有已支付意图、没有资源购买记录、商品仍可售且金额未漂移，则自动恢复为 `created` 再创建新二维码。
4. 服务端默认支付通道回落顺序改为支付宝优先，支付宝不可用时使用微信。

## 验收

1. 支付页自动轮询 3 次以上，二维码不闪烁，按钮文案不自动切换“正在刷新”。
2. 点击“取消支付”后二维码区域显示已关闭，取消按钮隐藏，订单摘要仍为待支付。
3. 点击“重新生成二维码”能直接生成新的微信或支付宝二维码，不再出现“订单当前不可支付”。
4. 商品页与支付页均默认支付宝在前、支付宝选中；关闭支付宝通道时自动回落微信。
5. 支付宝“立即购买”连续点击只发起一次下单/支付请求，按钮保持 loading 到跳转或错误返回。

## 回滚

1. 前端异常时可回滚本文件对应的 `checkout.vue` / `product.vue` 改动，后端支付意图接口保持兼容。
2. 后端异常时可回滚支付意图关闭逻辑，临时隐藏前端“取消支付”和“重新生成二维码”入口止血。
3. 如支付宝通道异常，可设置 `ALIPAY_PAY_ENABLED=false` 暂停支付宝，保留微信 Native。
